QUIC IETF

发表于 2020-09-08 | 浏览 次
字数统计: 2.5k

简介

QUIC具有以下特点：

  1. 流(Stream)的复用
  2. 流(Stream)和连接(Connection)的流控
  3. 低延迟的连接建立
  4. 支持连接迁移和resilience(弹性) to NAT rebinding
  5. 加密的header和payload
    本Draft关于QUIC的内容结构可以分为以下几类
  • Streams : basic service abstraction
  • Connections: the context in which QUIC endpoints communicate
  • Packets and Frames: basic unit used by QUIC to communicate
  • Enconding details

一些术语：

Address: When used without qualification, the tuple of IP version, IP address, UDP protocol, and UDP port number that represents one end of a network path.

Stream的核心概念

  • Streams can be unidirectional or bidirectional.
  • Streams can also be long-lived and can last the entire duration of a connection.
  • 流可以由任一端点创建，可以同时发送与其他流交错的数据，并且可以取消。QUIC不保证不同流的顺序。

    Stream的类型和标识

  • Streams are identified within a connection by a numeric value, referred to as the stream ID. Stream ID是一个62-bit的整数(0-2^62-1). 不同Connection的Stream ID可以相同，同一个Connection下的Stream ID唯一。 Stream ID在编码时是一个变长度的整数，(see Section 16).
  • Stream ID中最低位的比特标识流的发起/创建者，0标识是Client-initiated stream,1标识server-initiated stream.
    Stream ID中倒数第二位bit标识流是双向的还是单向的:0标识双向，1标识单向。
    故从低位标识角度看有4种流类型。
  • A stream ID that is used out of order results in all streams of that type with lower-numbered stream IDs also being opened.(乱序使用的流ID导致该类型的所有流以及编号较低的流ID也被打开。)

    发送和接收数据

  • STREAm frames分装来自已用的数据。Endpoint利用Stream frames中的Stream ID和Offset字段有序地存放数据。
  • QUIC不显示stream数据到达的有序性，但可以将out of order的数据交付给接收的应用。(便于音视频应用)
  • 一个endpoint可能接受到同一个stream上相同offset的数据多次，那么会丢弃已经接收到的数据。但该被多次发送的数据被进制篡改，否则接收的endpoint会认为connection error:PROTOCOL_VIOLATION.
  • 当数据传输，丢包后重传或在接收端传递给应用程序时，不会保留STREAM的帧边界。

    Stream优先级

  • Application可以向QUIC提供Stream的优先级信息。QUIC在为Streams分配资源时应该利用优先级信息。(但也就意味着应用层只提供优先级而无法参与进资源的分配调度)

    对Stream的操作

  • 发送时需要支持的操作
    • write data
    • end the stream: a STREAM frame with the FIN bit set.
    • reset the stream: a RESET_STREAM frame.
  • 接收时需要支持的操作
    • read data
    • 中止读取流并请求关闭，可能会导致一个STOP_SENDING frame
      应用层也需要嫩过去streams的状态改变，包括：对方开启或重置了一个stream,对方终止了stream的读取，新的数据到达，在当前的flow control下数据是否能写进stream帧。

      Stream的状态

      两个状态机：发送，接收
      对于双向流，打开流的条件稍微复杂些，因为发送侧或接收侧的打开都会导致流在两个方向上打开。
      一个Endpointd在open相同类型的stream时,stream ID必须是递增的。

      Stream发送状态机(Sending part)

      • 应用open一个stream,endpoint初始化stream type.
      • “Ready”代表一个新创建的stream可以接收来自appliation要发送的数据。
      • 发送第一个STREAM或者STREAM_DATA_BLOCKED 帧后进入”Send”状态。在实现中，知道大宋第一个STREAM帧时才分配stream ID并进入”SEND”状态，这允许更好的stream 优先级规划。
      • 如果Receiving part创建了，那Sending part会直接从”Ready”状态开始。
      • 还没有创建stream，则会在收到对方启动的双向流的发动部分以”Ready”状态开始。
      • 在”Send”状态下，endpoint传输或者重传STREAM帧。在此期间，endpoint遵守peer配置的流量控制约束，并接收和处理MAX_STREAM_DATA帧。endpoint会在”Send”状态下生成STREAM_DATA_BLOCKED 帧如果被streamh或者connection中的流量控制规则限制传输了。
      • 在所有的数据被发送完后，一个带有FIN的STREAM帧会被发送，并且sending part进入”Data Sent”状态。在这个状态中，endpoint只会重传stream data,并且endpoint不会遵守流控规则或者发送STREAM_DATA_BLOCKED帧。在此状态期间，endpoint可能会接收到MAX_STREAM_DATA帧，但endpoint会忽略掉。
      • 一旦所有的stream data被ACK，sneding part状态进入”Data Recvd”，终点状态。
      • 在”Ready”,”Send”或者”Data Sent”状态，应用可能会发出丢弃传输的指令。或者endpoint会接收到peer发送来的”STOP_SENDING”帧。这两种情况下，endpoint会发送一个”RESET_STREAM”帧，并会让状态进入”Reset Sent”状态。
      • 如果endpoint发动的第一帧就是”RESET_STREAM”，则sending part立即进入”Reset Sent”状态
      • 一旦”RESET_STREAM”帧被ACK,sending part会进入”Reset Recvd”,这也是一个终点状态。

        Stream接收状态机(Receiving part)

      • 被peer初始化的stream的receiving part当在接收到第一个STREAM/STREAM_DATA_BLOCKED或者RESET_STREAM帧时被创建。
      • 对于被peer初始化的stream来说，当sending part接收到MAX_STREAM_DATA或者STOP_SENDING帧也会创建receiving part.
      • receiving part的初始状态是”Recv”
      • 当endponint初始的stream的sending part进入”Ready”状态，receiving part进入”Recv”转态。
      • 一个endpoint会open一个stream当收到来自peer的该stream的”MAX_STREAM_DATA”帧或者”STOP_SENDING”帧。接收到一个unopened stream的”MAX_STREAM_DATA”帧表明peer已经open了该stream并提供流控机制。接收到一个unopened stream的”STOP_SENDING”帧表明peer不在愿意接收该stream上的数据。因为这两类帧都可能在STREAM/STREAM_DATA_BLOCKED帧前到达(丢包或者乱序)。
      • 在一个stream创建前，所有相同类型的有着更低的stream ID的stream都必须已经被创建了。这可以保证不同连接的endpoint创建的stream的一致性。
      • 在”Recv”状态下，endpoint会接收到STREAM和STREAM_DATA_BLOCKED帧。收到的数据会被bufferd并有序重组。当buffer中的数据被application取走，endpoint发送MAX_STREAM_DATA帧允许peer发送更多的数据。
      • 当收到一个带有FIN的STREAM帧时，stream的size变为已知。receiving part状态进入”Size Known”状态。在这个状态下，endpoint不在发送MAX_STREAM_DATA帧，只接收重传的数据。
      • 一旦所有的数据都到达，receiving 进入”Data Recvd”状态。在所有的数据都大大后STREAM和STREAM_DATA_BLOCKED帧会被丢弃。
      • “Data Recvd”状态会一直持续到所有的数据被递交给application。当所有的数据被递交完成时，stream状态进入”Data Read”状态，这是一个终点状态。
      • 当在”Recv”或者”Size known”状态接收到RESET_STREAM帧时，stream会进入”Reset Recvd”转态。这可能会打断给application的数据递交。
      • 当所有的数据都接收后(Data Recvd状态下)收到一个RESET_STREAM帧。在收到RESET_STREAM帧后可能还会收到stream data,在实现上可以自定义。
      • 没有要求规定当收到RESET_STREAM帧时禁止stream data的交付。是线上可以中断stream data的递交，丢弃掉那些没有被读取的，并ACK RESET_STREAM.如果stream data被完全接收并被缓冲以供application读取，则RESET_STREAM信号可能会被抑制或保留。 如果RESET_STREAM被抑制，则流的receiving part保留在“ Data Recvd”状态。
      • 一旦application获取到了stream被reset的信号，receiving part进入”Reset Read”状态，这是一个终点状态。

        允许的帧类型

      • stream发送方发送的帧中影响发送方或者接收方状态的有三种类型:STREAM,STREAM_DATA_BLOCKED,RESET_STREAM
      • 发送方在终点状态时禁止发送任何帧:Data Recvd或者Reset Recvd.
      • 发送方在Reset Sent转态或者终点状态时禁止发STREAM或者STREAM_DATA_BLOCKED帧
      • 接收方在任意状态下都可以接收这三种帧，因为延迟。STREAM,STREAM_DATA_BLOCKED,RESET_STREAM
      • stream接受方可以发送MAX_STREAM_DATA,STOP_SENDING帧。
      • 接收方在Recv状态下只能发送MAX_STREAM_DATA帧。
      • 接收方可能在没接收到RESET_STREAM帧的任意状态下发送STOP_SENDING帧。也就是除了Reset Recvd转态和Reset Read状态。有小概率在Data Recvd状态下发送STOP_SENDING帧。
      • 发送方可以再任意状态下收到这两种帧：MAX_STREAM_DATA,STOP_SENDING。

        双向stream的状态

      • 双向stream由sending part和receiving part组成。在实现上可以将双向流的状态表示为发送和接收流状态的组合。最简单的模型在sending part或receiving part处于非终端状态时将流显示为“打开”，而在发送和接收流都处于终端状态时将流显示为“关闭”。

        自发的状态转移

      • 如果应用不再要stream上接收到的数据，会发出一个error code、
      • 如果stream在Recv或者Size known状态下，应该发送一个STOP_SENDING帧来提议关闭stream.
      • 在发送STOP_SENDING帧后可能会有STREAM帧到达，这任然会被计入连接和流量控制。这些帧在接收到后会被丢弃。
      • STOP_SENDING帧会让对方发送一个RESET_STREAM帧。
      • endpoint应该从STOP_SENDING帧中复制error code到RESET_STREAM帧。endpoint在发送STOP_SENDING帧时可以忽略接收到的RESET_STREAM帧中的error code.
      • 如果STOP_SENDING帧在Data Sent转态下到达，endpoint会先发送RESET_STREAM帧再停止发送重传的STREAM帧。
      • STOP_SENDING SHOULD only be sent for a stream that has not been reset by the peer.
      • endpoint应该在STOP_SENDING帧丢失后在发送一个STOP_SENDING帧。如果所有的stream data都到达或者收到RESET_STREAM帧没有必要发送STOP_SENDING帧。
      • endpoint如果要终止双向流上所接收与发送，可以通过发送RESET_STREAM来终止一个方向，并发送STOP_SENDING帧促进在相反方向上的终止。

        flow control

        streams are flow controlled both individually and as an aggregate.
        接收方可以再任意时间在stream层级控制发送方能发送的最大数据量
        为了控制在connection里的并发量，endpoint可以控制其peer的最大初始stream数量

        Data flow control

        QUIC中有两个层级的data flow control
  1. stream flow control,防止单条stream消耗掉给一个connection分配的buffer，限制stream上的数据量。
  2. connection flow control, 防止发送方移除接受方的buffer容量，限制STREAM帧中总的stream data 字节数
    针对特定的stream用MAX_STREAM_DATA帧，针对connection用MAX_DATA帧。
